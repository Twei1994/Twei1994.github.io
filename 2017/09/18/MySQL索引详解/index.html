MySQL索引详解

前言

索引，是MySQL中很重要的一个知识点，在面试中也经常会被问到。之前对于索引的理解太粗浅，相关问题总是在死记硬背。现在仔细学习了下MySQL中索引的相关知识，发现许多问题都迎刃而解了。所以，学习知识到了一定程度，还是要从底层开始理解。

相关问题

这里列出几个有关于MySQL索引的问题：

  1. MySQL的索引有什么作用？可以分成几类？
  2. InnoDB的索引和MyISAM的索引有什么区别？
  3. InnoDB为什么要有主键？
  4. InnoDB的主键选哪种类型的字段性能最好？
  5. 组合索引什么时候生效，什么时候失效？
  6. 为什么使用like "%ABC%"索引会失效而like "ABC%"不会呢？

正文

为什么要使用数据库

要说索引，首先要从数据库说起。有没有想过，我们为什么要使用数据库？没有数据库的时候，大家再用什么存放数据？

首先回答第二个问题，如果不用数据库，还可以用什么存数据？我能想到的有以下几种方式：

  • 文本文件（.txt、.INI等）
  • XML、JSON文件
  • 电子数据表（Excel表）

它们各有优势，例如文本文档结构简单，XML和JSON文件对于分级数据比较友好，电子数据表可以进行某些简单的统计操作等。

说到底，不管以什么方式存放数据，我们的最终目标都是希望可以快速地对数据进行增、删、改、查操作，另外如果占用资源少就更好了。

在数据量比较小的时候，上面的这些方式看起来都还行。但是当数据量很大时，它们的缺点也很明显：

  1. 数据的存储会有大量冗余（浪费磁盘空间）；
  2. 查找（修改和删除的瓶颈也在查找）困难，速度很慢（需要一行一行的将文件读进来）；

为了解决这些问题，就得使用数据库。数据库可以帮我们将大量数据组织起来，按照某种方式存到计算机中。那么，数据库（这里指关系型数据库）是怎么解决上面的两个问题的呢？

  1. 通过多张表加上主键、外键的关联来解决数据存储冗余的问题；
  2. 通过增加索引来提高查询速度；

索引

那么，索引是什么？为什么能提高查找速度？

在关系数据库中，索引是一种单独的、物理的对数据库表中一列或多列的值进行排序的一种存储结构，它是某个表中一列或若干列值的集合和相应的指向表中物理标识这些值的数据页的逻辑指针清单。

通俗地说，索引就是目录。将数据库中的内容看成一本书，索引就是书的目录。我们要在字典里面查某个字，如“喵”，肯定不会一页一页地翻，看这一页有没有这个字，而是先看目录，找到拼音第一个是 m 的字的页码范围，然后在这个范围内继续找。

数据库的内容是存储在计算机的硬盘上的（如MyISAM的.frm、.MYD、.MYI文件）。我们使用数据库处理数据时，需要从硬盘中找到目标记录，并将它调入内存中。和字典类似，硬盘结构上也是分页的。它将所有的空间分成大小相等（可能是211到214字节）的页面，每次读取硬盘上的内容都以页为单位。

为了能够快速地从硬盘中得到存放目标数据的页，尽可能地减少对硬盘的访问，我们也需要一个目录，在数据库中，这个目录就被称为索引。

MySQL中的索引是通过B+树这种数据结构实现的。

从二叉排序树到B+树

我们都知道，二叉排序树（Binary Sort Tree/Binary Search Tree, BST）的查找效率很高，平均复杂度为O(logN)。但是当二叉排序树不平衡时（如最差时，仅有左子树或右子树），它的查找效率就变低了。如下图右边的BST中，我们要查找元素99，时间复杂度为O(N)。

所以我们希望二叉排序是平衡的，也就是每个节点的左子树和右子树的高度差至多等于1，这样的树就是平衡二叉树（Self-Blancing Binary Search Tree）。

如果我们的数据很多时，使用平衡二叉树时树的高度会很高。所以，通常会使用另外一种平衡二叉树的变种，多路查找树（Multi-way Search Tree），它的每个节点的孩子数可以有多个，并且每个节点可以存储多个元素，也被称为B树（这里的B指的是Blance，而不是Binary）。

这样，我们可以降低树的高度，通过高度为2~3的B树来存储所有的数据页（实际上，MySQL的B+树的高度也在2~3之间）。但是B树也不是最完美的，如下：

假设每个节点都存储在一个数据页上，当我们需要遍历这棵B树时，我们的路径是这样的：page2 -> page1 -> page3 -> page1 -> page4 -> page1 -> page5。我们每次切换节点时，都要重新载入根节点所在的数据页，这就不太好。有没有更高效的方式呢？

答案就是B+树，它能够实现高效地遍历树的所有元素。它的结构如下图所示：

它将根节点的关键字在叶子节点再次列出，并将所有的叶子节点都连接起来。B+数的叶节点包含了全部的关键字，并且按照顺序排列，非叶节点中存储着它的子树中最大（或最小）的关键字，仅作为索引使用。

B+树的好处就在于，如果是要随机查找，我们就从根结点出发，与B树的查找方式相同，只不过即使在分支结点找到了待查找的关键字，也只是用来索引，不提供实际记录的访问，还是需要到达包含此关键字的叶结点。

如果我们是需要从最小关键字进行从小到大的顺序查找，我们就可以从最左侧的叶子结点出发，不经过分支结点，而是延着指向下一叶子的指针就可遍历所有的关键字。所以B+树适合带有范围的查找。

InnoDB

不同于MyISAM，InnoDB的数据和索引存在一起，也就是它的主键索引是聚簇索引（Clustered Index）。

在使用InnoDB作为存储引擎时，我们需要指定一个主键。如果没有主键，但是有唯一索引（非NULL），就将该唯一索引作为主键。如果既没有指定主键也没有唯一索引，会自动生成一个长度为6字节的ROWID，将它作为主键（隐含，外部不可见）。所以InnoDB的表必须要有一个主键才行。

这是为什么呢？是因为在InnoDB中，数据文件本身要按主键聚集，表数据文件本身就是一棵B+树。这棵树的叶节点保存着所有的数据记录，并且是以表的主键构造的，因此InnoDB表必须有主键。如果没有主键，就不能存数据。

下图是InnoDB对上述数据的存储结构（以ID列为主键）：

可见看到，整个数据文件就是一棵B+树。要存储的每条记录按照主键值由小到大，存放在叶节点中，非叶节点使用主键值作为划分标准。这样在进行主键查询时，速度会很快。

辅助索引

在使用中，除了主键这一个索引之外，我们还会在其他字段上设置索引，加快以该字段为查询条件的SQL的执行效率。这些非主键字段上的索引被称为辅助索引（Secondary Key）。以第3列为例，增加索引，它的存储结构如下：

InnoDB的辅助索引与主键索引基本类似，不同之处在于它的叶节点中，保存的是主键值而不是整条记录。所以，在查询时，会分成两步：首先检索辅助索引获得主键，然后用主键到主索引中检索获得记录。

MyISAM

MyISAM用的是非聚簇索引（Unclustered Index），也没有必须要有主键的要求。它的数据文件放在.MYD中，索引文件放在.MYI中。MyISAM的索引也是采用B+树实现的，它的叶节点内容是一个指针，指向数据真正的位置。

在MyISAM中，主索引和辅助索引在结构上没有任何区别，只是主键索引要求字段值是唯一的，而辅助索引的字段值可以重复。假如在Score字段上建立一个辅助索引，则此索引的结构如下图所示：

问题解答

看到这里，博客开头的问题应该都可以解决了：

MySQL的索引有什么作用？可以分成几类？

答：MySQL索引的作用是为了提高数据查询的速度，也就是尽可能快得从磁盘上取出目标数据所在的页，采用是数据结构是B+树。它可以分成两类，一类是聚簇索引，另一类是非聚簇索引。

InnoDB的索引和MyISAM的索引有什么区别？

答：InnoDB的索引属于聚簇索引，MyISAM的索引属于非聚簇索引。

InnoDB为什么要有主键？

答：因为没有主键就没法存数据。（之前，总是以为，要先把数据存好，然后再根据需要加索引。对于MyISAM是这样，但InnoDB就不是先有数据，再加索引。）InnoDB的数据文件本身就是一个主键索引，它的建立必须依赖主键。所以，没有主键就没法存数据。实际上，如果没有指定主键，InnoDB会采用一个值唯一的字段来作为主键，如果没有唯一索引，则会自己生成一个隐含的ROWID来作为主键。

InnoDB的主键选哪种类型的字段性能最好？

答：最好是采用一个自增ID作为主键。有两个点：一是自增（也就是单调）；二是ID，也就是字段值尽可能小。

为什么要用自增（或者）单调的字段作为主键呢？这是由于B+树的结构的决定的，如下图所示，使用自增主键，每次添加新的记录时，直接添加到当前节点的后续位置即可，当前页如果满了，自动开辟一个新的页即可。这样就会形成一个紧凑的索引结构，近似顺序填满。由于每次插入时也不需要移动已有数据，因此效率很高，也不会增加很多开销在维护索引上。

如果不用自增主键，而是采用身份证或者学号等作为主键，那么每次插入位置不固定，近似于随机，如下图所示：

新纪录的位置是现有索引页中间的某个位置，MySQL不得不为了将新记录插到合适位置而移动数据，从而进行多次的读取磁盘操作，频繁地移动、分页会使磁盘产生碎片。而且索引结构也不够紧凑，后续不得不通过OPTIMIZE TABLE来优化。

为什么要用较小的字段值呢？因为InnoDB中，所有辅助索引都要引用主索引，过长的主索引会令辅助索引变得过大。

所以，很多时候，我们都会采用一个自增的ID来作为主键，虽然这个ID本身可能并没有什么实际意义。

组合索引什么时候生效，什么时候失效？

答：组合索引的生效条件是最左前缀。假如我对a，b，c三个字段添加了组合索引，那么当我按照（a,b,c）、（a,b）或（a）的组合来查询时，都会用到组合索引，其余情况则不会用到组合索引。

组合索引的存储情况如下所示：

组合索引还是一个B+树，不同的是组合索引键值的数量不是1，而是大于等于2。所以数据的排列是按照这个顺序：(1,1),(1,2),(2,1),(2,4),(3,1),(3,2)。因此。我们使用类似select * from t where a=xxx and b=xxx和select * from t where a=xxx时，都可以使用索引。但是我们只按b查询，如select * from t where b=xxx，是无法用到这个索引的，因为叶节点上b的排列是1,2,1,4,1,2，不是一个有序的。

这里还想说一下组合索引的另外一个好处：一般在一个SQL语句中，只会使用一个索引。但是如果SQL语句中既有查询又有排序时，只会使用查询字段的索引，然后进行一次filesort（可以通过explain来查看），这样的效率不高。这时，我们可以采用组合索引来对第二键值进行排序，也就是在查询和排序的字段上添加组合索引，这样可以省一次排序操作。

为什么使用like "%ABC%"索引会失效而like "ABC%"不会呢？

答：因为B+树是通过字段的值生成的，在使用时也是通过比较字段值和B+树节点来判断去往哪个节点。使用通配符%，也就意味着第一个字符是不确定的，所以无法和B+树节点进行比较，也就用不到索引了。类似于我们在成语字典查一个成语，如果不知道第一个字，也就无法使用目录了。

总结

这篇博客有这么几个知识点：

  1. MySQL中的索引是通过B+树实现的（不仅B+树，还有Hash等）；
  2. InnoDB的索引属于聚簇索引，必须要有主键；
  3. InnoDB的辅助索引的叶节点内容是主键值，所以还需要再查一次主键索引；
  4. MyISAM的索引属于非聚簇索引，主键索引和辅助索引没有区别；
