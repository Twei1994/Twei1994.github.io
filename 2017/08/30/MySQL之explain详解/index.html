MySQL之explain详解

前言

之前知道通过 explain 指令可以查看 SQL 查询语句的性能，但详细来说，每个字段表示什么意思，怎么说明 SQL 的性能，确实不太清楚。今天就认真学习下 explain 的用法。

正文

用法

EXPLAIN tbl_name
EXPLAIN [EXTENDED] SELECT select_options

前者可以得出一个表的字段结构等等，后者主要是给出该 SQL 语句相关的一些信息，这里讲述的是后者。下面是一个实例：

mysql> explain select user_id from dsp_user;
+----+-------------+----------+-------+---------------+---------+---------+------+------+-------------+
| id | select_type | table    | type  | possible_keys | key     | key_len | ref  | rows | Extra       |
+----+-------------+----------+-------+---------------+---------+---------+------+------+-------------+
|  1 | SIMPLE      | dsp_user | index | NULL          | PRIMARY | 4       | NULL |   37 | Using index |
+----+-------------+----------+-------+---------------+---------+---------+------+------+-------------+
1 row in set (0.00 sec)

字段值详解

下面解释下每个字段值的含义：

id

select 查询序列号，表示 SQL 语句执行的顺序，通常不重要。

select_type

查询的类型，主要是区别普通查询和联合查询、子查询之类的复杂查询。

常见取值有：

  • SIMPLE

它表示简单的 select，没有 union 和子查询；

  • PRIMARY

最外面的 select，在有子查询的语句中，最外面的 select 查询就是 primary；

  • NNION

union 语句的第二个或者说是后面那一个；

  • DEPENDENT UNION
  • NUION RESULT

table

输出的行所用的表。

type

连接所使用的类型，是较为重要的一个指标。

  • system

表仅有一行，是 const 类型的特例。不常见。

  • const

常量查询，在整个查询过程中这个表最多只会有一条匹配的行，用到了 primary key 或者unique 索引。

比如主键 id=1 就肯定只有一行，只需读取一次表数据便能取得所需的结果，且表数据在分解执行计划时读取。

mysql> explain select user_id from dsp_user where user_id =1;
+----+-------------+----------+-------+---------------+---------+---------+-------+------+-------------+
| id | select_type | table    | type  | possible_keys | key     | key_len | ref   | rows | Extra       |
+----+-------------+----------+-------+---------------+---------+---------+-------+------+-------------+
|  1 | SIMPLE      | dsp_user | const | PRIMARY       | PRIMARY | 4       | const |    1 | Using index |
+----+-------------+----------+-------+---------------+---------+---------+-------+------+-------------+
  • eq_ref

对于每个来自于前面的表的行组合，从该表中读取一行。

个人理解是在联合查询时，使用的条件是 UNIQUE 或 PRIMARY KEY。

mysql> explain select department from dsp_user,dsp_expert where dsp_user.user_id = dsp_expert.user_id;
+----+-------------+------------+--------+---------------+---------+---------+---------------------------+------+--------------------------+
| id | select_type | table      | type   | possible_keys | key     | key_len | ref                       | rows | Extra                    |
+----+-------------+------------+--------+---------------+---------+---------+---------------------------+------+--------------------------+
|  1 | SIMPLE      | dsp_expert | ALL    | PRIMARY       | NULL    | NULL    | NULL                      |   20 | NULL                     |
|  1 | SIMPLE      | dsp_user   | eq_ref | PRIMARY       | PRIMARY | 4       | dsplab.dsp_expert.user_id |    1 | Using where; Using index |
+----+-------------+------------+--------+---------------+---------+---------+---------------------------+------+--------------------------+
  • ref

对于每个来自于前面的表的行组合，所有有匹配索引值的行将从这张表中读取。

如果联接只使用键的最左边的前缀，或如果键不是 UNIQUE 或 PRIMARY KEY（换句话说，如果联接不能基于关键字选择单个行的话），则使用 ref。如果使用的键仅仅匹配少量行，该联接类型是不错的。

  • fulltext
  • ref_or_null

该联接类型如同 ref，但是添加了 MySQL 可以专门搜索包含 NULL 值的行。

在解决子查询中经常使用该联接类型的优化。

  • index_merge
  • unique_subquery

  • index_subquery

  • range

给定范围内的检索，使用一个索引来检查行。通常发生在在索引列上使用范围查询，如 >，<，in 等时，非索引列是 ALL。

mysql> explain select user_id from dsp_user where user_id in (1,2);
+----+-------------+----------+-------+---------------+---------+---------+------+------+--------------------------+
| id | select_type | table    | type  | possible_keys | key     | key_len | ref  | rows | Extra                    |
+----+-------------+----------+-------+---------------+---------+---------+------+------+--------------------------+
|  1 | SIMPLE      | dsp_user | range | PRIMARY       | PRIMARY | 4       | NULL |    2 | Using where; Using index |
+----+-------------+----------+-------+---------------+---------+---------+------+------+--------------------------+
  • index

按索引次序扫描，先读索引，再读实际的行，结果也是全表扫描，主要优点是避免了排序。（索引是排好序的，并且 all 是从硬盘中读的，index 可能不在硬盘上）

mysql> explain select user_id from dsp_user order by user_id;
+----+-------------+----------+-------+---------------+---------+---------+------+------+-------------+
| id | select_type | table    | type  | possible_keys | key     | key_len | ref  | rows | Extra       |
+----+-------------+----------+-------+---------------+---------+---------+------+------+-------------+
|  1 | SIMPLE      | dsp_user | index | NULL          | PRIMARY | 4       | NULL |   37 | Using index |
+----+-------------+----------+-------+---------------+---------+---------+------+------+-------------+
  • ALL

进行完整的表扫描。性能很差，通常可以增加更多的索引而不要使用 ALL，使得行能基于前面的表中的常数值或列值被检索出。

mysql> explain select user_id from dsp_user where user_name = "sss";
+----+-------------+----------+------+---------------+------+---------+------+------+-------------+
| id | select_type | table    | type | possible_keys | key  | key_len | ref  | rows | Extra       |
+----+-------------+----------+------+---------------+------+---------+------+------+-------------+
|  1 | SIMPLE      | dsp_user | ALL  | NULL          | NULL | NULL    | NULL |   37 | Using where |
+----+-------------+----------+------+---------------+------+---------+------+------+-------------+

结果值从好到坏依次是：

system > const > eq_ref > ref > fulltext > ref_or_null > index_merge > unique_subquery > index_subquery > range > index > ALL

一般来说，得保证查询至少达到 range 级别，最好能达到 ref。

possible_key

提示使用哪个索引会在该表中找到行，不太重要。

key

显示实际使用的键。如果是 NULL，表示没有索引被选择。

key_len

使用的索引长度。

ref

显示哪个字段或常数与索引 key 一起被使用。

rows

显示执行查询的行数（而不是返回的行数），数值越大越不好，说明没有用好索引。但对 InnoDB 不太准。

Extra

显示以上信息之外的其他信息，可能取值有以下几个：

  • Using index

此查询使用了覆盖索引（Covering Index），即通过索引就能返回结果，无需访问表。

mysql> explain select user_id from dsp_user;
+----+-------------+----------+-------+---------------+---------+---------+------+------+-------------+
| id | select_type | table    | type  | possible_keys | key     | key_len | ref  | rows | Extra       |
+----+-------------+----------+-------+---------------+---------+---------+------+------+-------------+
|  1 | SIMPLE      | dsp_user | index | NULL          | PRIMARY | 4       | NULL |   37 | Using index |
+----+-------------+----------+-------+---------------+---------+---------+------+------+-------------+
  • Using where

表示 MySQL 服务器从存储引擎收到行后再进行“后过滤”（Post-filter）。所谓“后过滤”，就是先读取整行数据，再检查此行是否符合 where 句的条件，符合就留下，不符合便丢弃。因为检查是在读取行后才进行的，所以称为“后过滤”。查询中含有 WHERE 子句时较常见。

mysql> explain select user_id from dsp_user where user_name = "twei";
+----+-------------+----------+------+---------------+------+---------+------+------+-------------+
| id | select_type | table    | type | possible_keys | key  | key_len | ref  | rows | Extra       |
+----+-------------+----------+------+---------------+------+---------+------+------+-------------+
|  1 | SIMPLE      | dsp_user | ALL  | NULL          | NULL | NULL    | NULL |   37 | Using where |
+----+-------------+----------+------+---------------+------+---------+------+------+-------------+
  • Using filesort

若查询所需的排序与使用的索引的排序一致，因为索引是已排序的，因此按索引的顺序读取结果返回，否则，在取得结果后，还需要按查询所需的顺序对结果进行排序，这时就会出现 Using filesort。

例如下面，user_id 是主键，所以它是索引。当 order by user_id 时，Extra 是 Using Index，而对别的列进行排序，就是 Using filesort，表示在查询之后，又进行了一次排序。

Using filesort（对非索引列排序）:

mysql> explain select user_id from dsp_user order by user_name;
+----+-------------+----------+------+---------------+------+---------+------+------+----------------+
| id | select_type | table    | type | possible_keys | key  | key_len | ref  | rows | Extra          |
+----+-------------+----------+------+---------------+------+---------+------+------+----------------+
|  1 | SIMPLE      | dsp_user | ALL  | NULL          | NULL | NULL    | NULL |   37 | Using filesort |
+----+-------------+----------+------+---------------+------+---------+------+------+----------------+

Using index（对索引列排序）:

mysql> explain select user_id from dsp_user order by user_id;
+----+-------------+----------+-------+---------------+---------+---------+------+------+-------------+
| id | select_type | table    | type  | possible_keys | key     | key_len | ref  | rows | Extra       |
+----+-------------+----------+-------+---------------+---------+---------+------+------+-------------+
|  1 | SIMPLE      | dsp_user | index | NULL          | PRIMARY | 4       | NULL |   37 | Using index |
+----+-------------+----------+-------+---------------+---------+---------+------+------+-------------+
  • Using temporary

需要创建临时表存储结果以完成查询。这种情况通常发生在查询时包含了 Group By 和 Order By 子句时或者联合查询时。

下面的例子显示了联合查询时出现的 Using temporary 的现象：

mysql> explain select dsp_user.user_id,department from dsp_user,dsp_expert where dsp_user.user_id = dsp_expert.user_id order by dsp_user.user_id; 
+----+-------------+------------+--------+---------------+---------+---------+---------------------------+------+---------------------------------+
| id | select_type | table      | type   | possible_keys | key     | key_len | ref                       | rows | Extra                           |
+----+-------------+------------+--------+---------------+---------+---------+---------------------------+------+---------------------------------+
|  1 | SIMPLE      | dsp_expert | ALL    | PRIMARY       | NULL    | NULL    | NULL                      |   20 | Using temporary; Using filesort |
|  1 | SIMPLE      | dsp_user   | eq_ref | PRIMARY       | PRIMARY | 4       | dsplab.dsp_expert.user_id |    1 | Using where; Using index        |
+----+-------------+------------+--------+---------------+---------+---------+---------------------------+------+---------------------------------+

一个有趣的现象是将上面的排序字段从 dsp_user.user_id 改成 dsp_expert.user_id 时，就不会使用临时表了。

mysql> explain select dsp_user.user_id,department from dsp_user,dsp_expert where dsp_user.user_id = dsp_expert.user_id order by dsp_expert.user_id;
+----+-------------+------------+--------+---------------+---------+---------+---------------------------+------+--------------------------+
| id | select_type | table      | type   | possible_keys | key     | key_len | ref                       | rows | Extra                    |
+----+-------------+------------+--------+---------------+---------+---------+---------------------------+------+--------------------------+
|  1 | SIMPLE      | dsp_expert | index  | PRIMARY       | PRIMARY | 4       | NULL                      |   20 | NULL                     |
|  1 | SIMPLE      | dsp_user   | eq_ref | PRIMARY       | PRIMARY | 4       | dsplab.dsp_expert.user_id |    1 | Using where; Using index |
+----+-------------+------------+--------+---------------+---------+---------+---------------------------+------+--------------------------+

原因在于 MySQL 在联合查询时，会有一个驱动表的概念，对驱动表可以直接排序，对非驱动表（的字段排序）需要对循环查询的合并结果（临时表）进行排序。在 explain 的结果中，处于上面的表是驱动表（这个例子中是 dsp_expert，所以对它的字段排序不用使用临时表）。

当进行多表连接查询时， [驱动表] 的定义为：

1）指定了联接条件时，满足查询条件的记录行数少的表为[驱动表]；

2）未指定联接条件时，行数少的表为[驱动表]。

  • Distinct
  • Not exists
  • range checked for each record

总结

  1. explain 的四个重要字段：type、key、rows、extra；

  2. 如果 type 的值为 index 或者 ALL，那么说明该 SQL 性能一般，需要优化；

  3. 如果 key 的值为 NULL，说明该 SQL 没有使用索引，可以考虑在关键字段上增加索引；

  4. row 的值代表了进行本次查询时，搜索记录的条数，当这个值特别大的时候，说明该 SQL 语句性能差；

  5. 如果 Extra 字段的值为 Using filesort 或 Using temporary，也是需要优化的，可以通过调整 order by 或者 group by 的字段来实现；

  6. 联合查询时，一定要多用 explain 来查看查询性能；
