进程与线程的区别

前言

进程和线程，是操作系统原理中的基本概念，也是面试中经常被问到点。这篇博客是我在看了许多博客基础上总结的，由于我没有系统的学过《操作系统原理》，所以理解上可能有偏差。大家觉得有什么问题，请给我发邮件，一起讨论。

我的邮件：tjutwei@163.com

正文

定义

进程和线程简单的定义如下：

  1. 进程：是程序的一次执行活动；
  2. 线程：是进程中的最小执行单元；

下面是详细的解释：

进程（Process）是计算机中具有一定独立功能的程序关于某数据集合上的一次运行活动，是系统进行资源调度和分配的基本单位，是操作系统结构的基础；

线程（Thread）是比进程小的能独立运行的执行单位，是 CPU 调度和分配的基本单位。

深入理解

相信大家看了以上的定义，还是不太明白吧。接下来我们从一个宏观的层面上来理解：

首先，计算机的 5 大组成部分我们都不陌生——控制器，运算器，存储器，输入设备和输出设备。这 5 大组成都是抽象的概念，对应到具体的硬件上就是：

CPU = 控制器 + 运算器

内存 + 硬盘 = 存储器

主板 = I/O 总线，输入输出系统

键盘，鼠标，扫描仪，显示器等等 = I/O 设备

而我们所说的程序，它的定义是这样的：指为了得到某种结果而可以由计算机等具有信息处理能力的装置执行的代码化指令序列，或者可以被自动转换成代码化指令序列的符号化指令序列或者符号化语句序列。简而言之，就是我们写的代码。

我们代码想要执行，需要使用计算机的一些资源。除去输入、输出设备，这些资源主要可以分成两类：计算资源和存储资源，也就是电脑的 CPU 和 内存（RAM），这两样东西是我们程序运行不可或缺的。在执行程序时，需要向计算机申请。

也许你要问了，我们写代码时，并没有申请过这些东西啊？只是写好代码，点击运行就行了。确实，你没有自己做这件事，那是因为别人替你做过了。这个好心人就是操作系统。

我们 CPU 和 内存两个方面出发，来学习操作系统的作用，最终来理解进程和线程。

CPU和调度器

在单核计算机里，CPU 是无法被多个程序并行使用的，也就是同一时间，只有一个程序能占用 CPU。没有操作系统的情况下，一个程序一直独占着 CPU，直到它运行结束。

当有多个程序，但只有一个 CPU 时，怎么办呢？是大家轮流着来，执行完 A 再执行 B？还是 A 先执行 1 分钟轮 B，然后 1 分钟后再轮 A？

这个获取 CPU 的策略就是所谓的 “进程调度”。当然，我们还没讲到进程，先忽略它。总而言之，多个程序共享一个 CPU 时，需要程序员出手调节。程序员安排好运行计划 – 某时刻 CPU 由 A 来独享，下一时刻 CPU 由 B 来独享，或者其他类似的计划。

这种安排计划后来成为操作系统的一个核心组件，被单独名命为 Scheduler，即调度器，它关心的只是怎样把单个 CPU 的运行拆分成一段一段的“运行片”，轮流分给不同的程序去使用，而在宏观上，因为分配切换的速度极快，就制造出多程序并行在一个 CPU 上的假象。

内存和虚拟内存

现在 CPU 分配的任务解决了，该考虑内存的分配了。内存不像 CPU，它可以被同时多个程序共用。

也许你觉得这样就不需要分配了，大家一起用就好了。但是这样是有问题的，例如：

内存覆盖

程序 A 申请了一块内存，用于存放自己的数据，并且在获得 CPU 的时候运行了一会儿。这时，调度器通知它，CPU 现在得给 B 用了，你先等着吧。此时 A 就得记录下自己的执行情况和中间变量，然后让出 CPU。但是 B 申请内存时，和 A 重叠了，当它进行运算时，覆盖了 A 保存的内容。当再次轮到 A 使用 CPU 时，A 懵逼了，它根本不知道上次运行到哪了，所以就崩溃了。

当然，这个问题好解决，就是在第一开始，程序员就手工为每个程序安排运行的空间 – A 使用物理地址 0x00-0xff 的内存块，B 使用物理地址 0x100-0x1ff 的内存块，等等。

内存浪费

给每个程序分配一块内存，并保证它们不重复，确实解决了内存覆盖的问题，但是这样的效率并不高。因为在分配内存时，要分配给它所需要的最大内存，但这个程序并不是时刻都需要这么大的内存。而它的空闲内存因为已经有主人了，所以不能分配给其他的程序。这就造成了浪费。

所以，我们需要一个内存分配的策略，来确保每个程序都有合适的内存可用。“虚拟内存”就是为了解决这个问题引入的，它分配给程序“一个连续完整的地址空间”，但实际上这个地址空间是被分隔成多个物理内存碎片，还有部分暂时存储在外部磁盘存储器上，在需要时进行数据交换。

为了实现这个功能，在硬件上，CPU 增加了一个专门的模块叫 MMU（Memory Management Unit），负责转换虚拟地址和物理地址；在操作系统上，增加了另一个核心组件：Memory Management，即内存管理模块，它管理物理内存、虚拟内存相关的一系列事务。

具体的内存分配过程是这样的：每个程序在启动和执行时，会被分配到一块虚拟内存空间。这个虚拟地址空间是“完全一样”的，经由操作系统和硬件 MMU 协作，会映射到不同的物理地址空间上。

注：

  1. 不同的程序对应不同的的物理内存空间，彼此独立，不用一些特殊手段，无法访问别的程序的物理内存；
  2. 虚拟内存完全一致，是指这段虚拟内存空间的起点和终点都一样。例如在 32 位的系统中，分配的虚拟内存空间大小为 4GB（2^32字节），它起始于 0x00000000 结束于 0xFFFFFFFF；
  3. 虚拟内存通过下面两个方式解决内存浪费问题：
    • 不是每个程序都分配 4GB 的物理内存，而是按需为每个程序分配，4GB 是 32 位系统中一个程序所能占空间的上限；
    • 将程序所需的内存划分为 4KB 大小的内存页，并根据使用情况将这些内存页存储在硬盘上或加载到 RAM 中；

进程和线程

总结上面的内容，就是通过调度器和虚拟内存，每个程序启动和执行都可以获取到所需要的内存和 CPU。

人们把上面这句话中的“程序启动和执行”概括成了一个词，叫做进程。进程就表示程序的执行过程，所以上面那句话可以这样说：通过调度器和虚拟内存，每个进程都可以获取到所需要的内存和 CPU。

现在，不同的程序，可以不关心底层的物理内存分配，也不用关心 CPU 的协调共享。但是又出现了一个问题：在程序执行的时候，怎么尽可能高效地利用系统分配给它的内存？怎样才能利用进程的空闲内存呢？这个问题的本质就是如何共享相同的内存。

上面说到，每个进程的内存都是相互独立的，不能彼此访问，所以共享相同的内存只能在进程内部解决。这时候，一个叫线程的模型就出现了，它们被包裹在进程里面，在调度器的管理下共享 CPU，拥有同样的虚拟地址空间，同时也共享同一个物理地址空间，然而，它们无法越过包裹自己的进程，去访问别的进程的地址空间。

总结

不管是进程还是线程，描述的都是一段代码的执行过程，不同的是它们代表的计算机资源分配和调度方式。

可以共享内存的方式是线程，内存独立的方式是进程；进程的代码包括整个程序，线程的代码可能是这个程序的一部分；CPU 是操作系统分配给进程的，但具体执行是在线程上。

注：在有的操作系统里，进程不是调度单位（即不能被调度器使用），线程是最基本的调度单位，调度器只调度线程，不调度进程，比如 VxWorks

常见问题

进程与线程的区别

  1. 最大的区别就是对系统资源的占用。进程拥有比较多的系统资源，而线程基本上不拥有系统资源，只拥有一点在运行中必不可少的资源（如程序计数器，一组寄存器和栈），但是它可与同属一个进程的其他的线程共享进程所拥有的全部资源；
  2. 线程不能独立运行，它依赖于进程，进程不依赖线程；
  3. 一个程序至少包含一个进程，一个进程至少包含一个线程；
  4. 进程有独立的地址空间，一个进程崩溃后，在保护模式下不会对其它进程产生影响，而线程挂掉就等于整个进程挂掉，所以多进程的程序要比多线程的程序健壮；

进程与程序的区别

程序是指令和数据的有序集合，其本身没有任何运动的含义，是一个静态的概念，而进程则是在处理机上的一次执行过程，它是一个动态的概念。

进程的执行离不开程序，在给进程分配的虚拟地址空间中，有一个叫做代码区，存放着程序的代码。

注：分配给每个进程的内存可以分成 4 个区：栈区、堆区、数据区和代码区。

栈区：主要用来存放局部变量, 传递参数, 存放函数的返回地址；

堆区：用于存放动态分配的对象, 当使用 malloc 和 new 等进行分配时，所得到的空间就在堆中。动态分配得到的内存区域附带有分配信息, 所以可以 free 和 delete 它们；

数据区：全局，静态和常量是分配在数据区中的，数据区包括 bss（未初始化数据区）和初始化数据区。
