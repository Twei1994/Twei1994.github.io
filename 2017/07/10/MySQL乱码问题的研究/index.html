MySQL乱码问题的研究

前言

数据库的乱码问题一直很让人头疼，其实根本的原因在于对字符集和数据库存储过程理解不够。这两天把MySQL的乱码问题研究了一下，在这里记录下收获和感想。

字符集

解决乱码问题，首先要理解字符集。常见的字符集有 ASCII，GBK，GB2312，Unicode，UTF-8等。关于字符集的详细知识可以看我之前写的这篇博客。

总而言之，因为计算机只能处理 0 和 1，所以不管是英文字母、汉字、日文或者是其他的一些符号，都要用 0 和 1 来表示。这些表示方法就是字符集。不同时期和不同机构定义的字符集内容和规则是不一样的。

例如：对于同一个汉字 “中”，在 Unicode 字符集中的编码是 01001110 00101101，而在 UTF-8 中的编码是 11100100 10111000 10101101。

一个二进制的字符串，如果没有告诉我们对应的字符集，那么我们是无法知道它到底表示了哪个字符。采用了错误的字符集对这个二进制的字符串进行解释，就会造成乱码。

MySQL中的乱码问题

存储过程

我们首先想一下用户存储一条记录的过程，首先在 client 端输入一条 SQL 语句，然后通过 TCP/IP 传递给 MySQL Server 进程，然后 Server 端执行 SQL 语句，并将记录放在某个文件中。

基本每个过程都要涉及到字符编码。从 MySQL 关于的字符集设置的系统变量就能看出来。

MySQL字符集系统变量

通过执行 show variables like "character%" 命令可以看到 MySQL 中所有关于字符集的变量，如下图所示：

它们对应的含义如下：

变量名 含义
character_set_server 默认的内部操作字符集
character_set_client 客户端来源数据使用的字符集
character_set_connection 连接层字符集
character_set_results 查询结果字符集
character_set_database 当前选中数据库的默认字符集
character_set_system 系统元数据(字段名等)字符集

在新建数据库（Database）、数据表（Table）、字段（Column）时，这些参数就会产生作用：

  • 新建数据库时，若未明确指定字符集，则采用 character_set_server 指定的字符集；
  • 新建数据表时，若未明确指定字符集，则采用当前库所采用的字符集；
  • 新建，修改表字段时，若未明确指定字符集，则采用当前表所采用的字符集。

最终一条记录的字符集是由它所在的字段决定的。

字段的字符集信息可以用命令 show full columns from table_name 来查看。

更新（insert，update）和查询（select）时字符集的转换

更新时字符集转换过程： character_set_client –> character_set_connection –> 表字符集；

查询时字符集转换过程：表字符集 –> character_set_result

这里其实还存在另外一个过程，就是SQL的输入和结果的显示。因为我们要通过输入输出设备来输入 SQL 语句和获取查询结果，例如 CMD 窗口，CMD 窗口在显示时也会使用字符（windows 默认的是 GBK），所以它的字符集也需要我们考虑在内。通过设置可以修改，具体方法可以查看这里。

字符集转换原理

转换原理并不复杂。每个字符集可以看作是一个表，存储着字符和对应的编码。假设 GBK字符集的字符串“小明”，需要转为 UTF8 字符集存储，实际就是对于“小明”字符串中的每个汉字去 UTF8 编码表里面查询对应的二进制，然后存储。

实验

对常见的情况进行了实验，下面先给出结果图：

这个实验分成两个部分，分别在 CMD 编码为 GBK 和 utf8 下进行了实验。每个实验分为插入和查询两个部分。names 是指执行 set names character_set后的字符集，它会修改 character_set_client，character_set_connection 和 character_set_results 这三个系统变量。

Result 一栏中，绿色表示结果正确，黄色表示查询结果为乱码，红色表示插入出错。

结论

  • 在插入操作时，如果 client（这里指 CMD）的字符集和 character_set_client 的不一致，操作会出错，错误代码是 1366 或者 1406；

  • 在查询操作时，如果 client（这里指 CMD）的字符集和 character_set_result 的不一致，会产生乱码；

第2和第4行的查询结果。

倒数第2和倒数第4行的查询结果。

它们的共同特点就是在插入和查询过程中，共进行了一次字符集转换。这样在结果的显示时，返回的字符集和显示工具的字符集不同，从而导致了乱码。

  • 正确执行的共同特点就是在插入和查询过程中，共进行了零次或两次字符集转换。这样返回结果和显示的字符集相同，就不会产生乱码。

只要做到以下两点，就不会产生乱码（其实与表的字符集没有关系）：

  1. 保证输入时的字符集和 character_set_client 一致，保证输出时的字符集和 character_set_result 一致；
  2. 要通过 set names character_set 来统一修改字符集，保证character_set_client、character_set_connection、character_set_result 这三者是一致的。

字符集相关的SQL命令

1.查看字符集编码设置

SHOW VARIABLES LIKE ‘%CHARACTER%’

2.设置字符集编码

SET NAMES xxx；

这个语句相当于设置了3个系统变量，character_set_client，character_set_connection 和 character_set_results。

3.修改数据库字符集

ALTER DATABASE  DATABASENAME  CHARACTER SET XXX;

这个语句只修改库的字符集，影响后续创建的表的默认定义；对于已创建的表的字符集不受影响。

4.修改表的字符集（不确定，找时间确认一下）

ALTER TABLE TABLENAME CHARACTER SET XXX；

这个语句只修改表的字符集，影响后续该表新增列的默认定义，已有列的字符集不受影响。

ALTER TABLE TABLENAME CONVERT TO CHARACTER SET XXX;

这个语句同时修改表字符集和已有列字符集，并将已有数据进行字符集编码转换。

5.修改列字符集

ALTER TABLE `TABLE_NAME` MODIFY COLUMN `COLUMN_NAME`  CHARACTER SET xxx

6.查询字符的十六进制和长度

SELECT HEX(COL_NAME) FROM TABLE_NAME;    // 十六进制显示

SELECT LENGTH(COL_NAME) FROM TABLE_NAME;  // 字节数

其它

乱码一般都是中文或者其他特殊符号，数字和英文不会产生乱码。这是因为包含数字和英文字母的字符集 ASCII 产生的最早，其它的字符集都会包含它。
