MySQL优化

前言

接触MySQL也挺长时间了，但基本停留在使用的层面，这里就稍微深入地探讨一下MySQL优化方面的知识。

MySQL优化的原则有这几个方面：减少系统的瓶颈、减少资源的占用、加快系统的响应速度。

例如，通过优化文件系统，提高磁盘I/O的读写速度；通过优化操作系统调度策略，提高MySQL在高负荷情况下的负载能力；通过调整表结构和查询语句、增加索引等提高系统响应速度。

正文

个人认为MySQL的优化可以分成3个方面：数据库设计和结构优化、MySQL服务器和配置优化、查询优化。

其中，索引的使用是查询中很重要的一个方面，所以我把它单独列出来作为一个优化项。

下面从这个4个方面来说一下MySQL中常用的优化方法。

MySQL服务器和配置优化

硬件

  • 配置较大的内存

内存的速度要比磁盘I/O快得多，所以可以给MySQL配置较大的内存，增加缓冲区的容量，让更多的查询结果尽可能长时间地停留在内存中，从而减少磁盘I/O。

  • 选择高速磁盘系统

更快的转速、RAID、阵列卡等，减少读盘的等待时间，提高响应速度。

  • 合理分布磁盘I/O

将磁盘I/O分散在多个设备上，以减少资源竞争，提高并行操作能力。

  • 选择多处理器系统

选择多核CPU、主频高的CPU，MySQL是多线程的数据库，多个CPU可以同时执行多个线程。

  • 网络环境选择

尽量部署在局域网、SCI、光缆、千兆网、双网线提供冗余、0.0.0.0多端口绑定监听。

操作系统内核（不太理解）

  • 使用64位的操作系统，更好地使用大内存；
  • 设置noatimenodiratime；
  • 优化内核参数；
  • 加大文件描述符限制；
  • 文件系统选择 xfs；

MySQL参数配置

MySQL的配置参数都在my.ini或则my.cnf文件中。

一些对性能影响较大的参数如下所示，包括I/O处理的常用参数；最大连接数设置；缓存使用参数的设置；慢日志的参数的设置；InnoDB相关参数的设置等。

key_buffer_size
MySQL索引缓冲区的大小。如果是采用MyISAM的话要重点设置这个参数，根据（key_reads/key_read_requests）判断

max_connections
服务器允许的最大连接数，尽量不要设置太大，因为设置太大的话容易导致内存溢出

wait_timeout
线程连接的超时时间，尽量不要设置很大，推荐10s

thread_concurrency
线程并发利用数量，(cpu+disk)*2，根据(os中显示的请求队列和tickets)判断

sort_buffer_size
排序缓冲区的大小。该值越大，进行排序的速度越快

read_rnd_buffer_size
当根据键进行分类操作时获得更快的--ORDER BY

join_buffer_size
join连接使用全表扫描连接的缓冲大小，根据select_full_join判断

read_buffer_size
全表扫描时为查询预留的缓冲大小，根据select_scan判断

tmp_table_size
临时内存表的设置，如果超过设置就会转化成磁盘表，根据参数(created_tmp_disk_tables)判断

innodb_buffer_pool_size
InnoDB表数据和索引的最大缓存。根据（hit riatos和FILE I/O）判断

innodb_log_file_size(默认5M)
InnoDB引擎的redo log文件，设置较大的值意味着较长的恢复时间

innodb_flush_log_at_trx_commit(默认1)
表示何时将缓冲区数据写入日志，并将日志文件写入磁盘。
1. 0表示每秒进行一次log写入cache，并flush log到磁盘；
2. 1表示在每次事务提交后执行log写入cache，并flush log到磁盘；
3. 2表示在每次事务提交后执行log写入cache，每秒执行一次flush log到磁盘；

数据库设计和结构优化

数据表尽量符合范式

  • 1NF：表的列的具有原子性，不可再分解，即列的信息，不能分解成为更小的数据项；
  • 2NF：表中的记录是唯一的，通常通过设计一个主键来实现；
  • 3NF：表中不要有冗余数据，也就是说，如果某个字段能够被其它字段推导出来，就不应该存在；

当然，没有冗余的数据库未必是最好的，有时为了提高运行效率，就必须降低范式标准，适当保留冗余数据。

合理选择存储引擎

在开发中，经常使用的存储引擎有MyISAM/InnoDB/Memory。

MyISAM： 数据库并发不大，读多写少，sql 语句比较简单，对事务要求不高，比如 bbs 中的发帖表，回复表；

InnoDB：并发访问大，写操作比较多，有外键、事务等需求的应用，系统内存较大。比如订单表，账号表；

Memory：数据不需要入库，同时又频繁的查询和修改，速度极快；

字段数据类型选择

字段数据类型的选择的一般原则：

  • 根据需求选择合适的字段类型，在满足需求的情况下字段类型尽可能小；
  • 只分配满足需求的最小字符数，不要太慷慨；

原因：更小的字段类型更小的字符数占用更少的内存，占用更少的磁盘空间，占用更少的磁盘 IO 以及占用更少的带宽。

CHAR 和 VARCHAR 的选取

它们的区别在于：CHAR(M) 类型的数据列里，每个值都占用 M 个字节，如果某个长度小于 M，MySQL 就会在它的右边用空格字符补足；VARCHAR(M) 类型的数据列里，每个值只占用刚好够用的字节再加上一个用来记录其长度的字节（即总长度为 L+1 字节）。

二者的选取主要依据如下原则:

  1. 如果列数据项的大小一致或者相差不大，则使用 CHAR；
  2. 如果列数据项的大小差异相当大，则使用 VARCHAR；
  3. 对于 MyISAM 表，尽量使用 CHAR。 因为 VARCHAR 类型随着修改容易造成磁盘碎片，使用 CHAR 的缺点就是占用磁盘空间；
  4. 对于 InnoDB 表，从减少空间占用量和减少磁盘 I/O 的角度，使用 VARCHAR;
  5. 表中只要存在一个 VARCHAR 类型的字段，那么所有的 CHAR 字段都会自动变成 VARCHAR 类型，因此建议定长和变长的数据分开。

编码选择

  • 单字节 latin1
  • 多字节 utf8 (汉字占 3 个字节，英文字母占用一个字节)

如果含有中文字符的话最好都统一采用 utf8 类型，避免乱码的情况发生。

NULL OR NOT NULL

尽可能设置每个字段为 NOT NULL，除非有特殊的需求，原因如下：

  1. 使用含有 NULL 列做索引的话会占用更多的磁盘空间，因为索引 NULL 列需要而外的空间来保存；
  2. 进行比较的时候，程序会更复杂；
  3. 含有 NULL 的列 SQL 难优化，如果是一个组合索引，那么这个 NULL 类型的字段会极大影响整个索引的效率。

读写分离

开启 MySQL 复制，实现读写分离、负载均衡，将读的负载分摊到多个从服务器上，提高服务器的处理能力。

分表和分区

当表存储了百万级乃至千万级条记录时，在查询和插入的时候耗时太长，性能低下，如果涉及联合查询的情况，性能会更加糟糕。

分表和分区的目的就是减少数据库的负担，提高数据库的效率。

分表可以分成水平拆分和垂直拆分两种，水平拆分就是将一张大表拆分成多张小表，每张小表的结构与原表相同，只是记录条数要少；垂直拆分就是将大表中的某些字段（使用频率低）取出来，放在另一张表中。

查询优化

COUNT(*)

在 MyISAM 下没关系，但尽量不在 InnoDB 下使用。

如果要在 InnoDB下使用，建议另外弄一张统计表，采用 MyISAM，定期做统计。一般的对统计的数据不会要求太精准的情况下适用。

尽量不在SQL中做计算

Mysql 的作用是用来存取数据的，不是做计算的。 在 SQL 中做计算很耗资源，尽量取出数据在其它地方做计算。

尽量避免 SELECT *

用哪些字段就取哪些字段。在访问量大或者记录中某个字段数据量大时，SELECT * 在执行效率、IO 资源以及网络带宽方面，会造成严重的浪费。

用连接 JOIN 代替子查询

因为使用 join，不需要在内存中创建临时表。

已知查询行数时用 LIMIT

查询尽可能使用 LIMIT 减少返回的行数，减少数据传输时间和带宽浪费。

避免隐式类型转换

例如字符型一定要用引号括起来，数字型一定不要使用引号，如： SELECT score FROM student WHERE name = 'twei' AND id = 5;

SQL 关键词大写

所有的 SQL 关键词用大写，养成良好的习惯，避免 SQL 语句重复编译造成系统资源的浪费。

开启慢查询

开启慢查询，定期用 EXPLAIN 优化慢查询中的 SQL 语句。

ANALYZE、CHECK、OPTIMIZE

经常用 ANALYZE TABLE 和 CHECK TABLE 来分析和检查表，用 OPTIMIZE TABLE 来整理碎片。

增加中间表

对于经常要做联合查询的表，可以增加一张中间表，存放联合查询的数据，牺牲存储空间加快查询速度。

优化插入数据（INSERT）

在插入数据时，可以通过以下几种方式来提高插入速度：

  • 禁用索引

在插入数据时，MySQL会同时更新表的数据和索引。如果插入大量数据，建立索引会降低插入速度。所以，可以在插入数据前关闭索引，插入完成后再开启索引。

关闭索引的语句如下：ALTER TABLE table_name DISABLE KEYS
打开索引的语句如下：ALTER TABLE table_name ENABLE KEYS

  • 禁用唯一性检验

在插入数据时，MySQL也会进行唯一性检查。与索引一样，唯一性检验也会降低插入速度。可以在插入前关闭，然后再插入后打开。

禁用唯一性检验的语句如下：SET UNIQUE_CHECKS=0
开启唯一性检验的语句如下：SET UNIQUE_CHECKS=1

  • 使用批量插入

批量插入的速度要比逐条插入快。

INSERT INTO table_name VALUES (1, a, 80),(2, b, 90),(3, c, 34);

的速度要快于下面这三句的执行速度:

INSERT INTO table_name VALUES (1, a, 80);
INSERT INTO table_name VALUES (2, b, 90);
INSERT INTO table_name VALUES (3, c, 34);

索引优化

MySQL 中最常用到的优化方式就是添加索引，在需要查询的字段上添加索引可以大大加快查询的速度。索引采用 B-TREE，HASH，R-TREE 等数据结构，本质上就是记录着字段值和存储位置对应关系的文件，利用它可以很快地找字段为特定值的记录所在的位置，而不需要扫描整个数据表。

MySQL 中的索引有以下几种：主键索引，唯一索引，全文索引，普通索引，组合索引。

通过 EXPLAIN 指令查看该 SQL 语句中的索引使用情况。

适合创建索引的字段

  • 在 WHERE 和 ORDER BY 条件中出现；
  • 该字段的内容不是唯一的几个值（sex）；
  • 该字段内容不是频繁变化；

为多个查询字段添加组合索引

例如，经常选择满足某时间 time 和某地点 city 的记录时，在 time 和 city 上添加组合索引。

尽量使用短索引

对字符串字段添加索引时，如果可能应该指定一个索引长度。短索引不仅可以提高查询速度而且可以节省磁盘空间和 I/O 操作。

例如，如果有一个 CHAR(255) 的列，如果在前 10 个或 20 个字符内，多数值是惟一的，那么就不要对整个列进行索引。

索引失效

一些会导致索引失效的操作：

  • LIKE 和 ERGEXP，在搜索模板的第一个字符是通配符时，如 LIKE "%aaa%" 不会使用索引，而 like "aaa%" 会使用索引；
  • WHERE 子句的查询条件里有不等号 != 或 <>，如 WHERE column != xxx；
  • WHERE 子句的查询条件里有 or，如 WHERE column1 = xxx or column2 = xxx，此时可以采用 union all` 来替代；
  • WHERE 子句的查询条件里使用了函数，如 WHERE DAY(column) = xxx；
  • WHERE 子句中对字段进行 NULL 值判断，如 WHERE num IS NULL;
  • WHERE 子句的查询条件里有 in 或 not in，如 WHERE id IN (1,2,3);，此时用 between 代替；
  • 如果使用复合索引，那么必须使用到该索引中的第一个字段作为条件时才能保证系统使用该索引，否则该索引将不会被使用，并且应尽可能地让字段顺序与索引顺序相一致；
  • JOIN 操作中，主键和外键的数据类型不同；
